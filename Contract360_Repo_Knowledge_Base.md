# Contract360 GraphRAG Repository Knowledge Base

> **Purpose:** This document is a complete engineering handoff / knowledge base for the fresh Contract360 GraphRAG repository. It explains the repository structure, ingestion pipeline, Azure AI Search indexing, knowledge graph pipeline, RAG/query routing, Streamlit frontend, current demo scope, known limitations, and next-step architecture guidance for an AI assistant or teammate continuing development.

---

## 1. Executive Summary

This repository implements a proof-of-concept Contract360 assistant for legal/contract question answering.

The system currently supports two major capabilities:

1. **Search RAG over parsed contract trees**
   - Contract files are parsed into a tree structure.
   - Tree nodes are chunked into retrieval-ready documents.
   - Embeddings are created.
   - Documents are uploaded to Azure AI Search.
   - Users can ask semantic/search questions over indexed contracts.

2. **GraphRAG over one graph-enabled demo contract**
   - The Edison contract tree is normalized into a structural KG.
   - Legal semantic facts are extracted from selected clauses.
   - The graph is stored in Cosmos DB Gremlin.
   - Azure AI Search documents are enriched with `kgId` to connect search results to graph nodes.
   - A router chooses graph, search, or hybrid retrieval.
   - Final answers are generated by Azure OpenAI using retrieved context only.

The current graph-enabled contract is:

```text
Edison_NYPA_OandM_Contract_1
```

Newly uploaded contracts through Streamlit are currently:

```text
Search-only
```

Graph upsertion and semantic legal extraction for uploaded contracts is intentionally deferred until after the demo.

## 2. Current End-to-End Architecture

```text
PDF / TXT / MD
   ↓
Document Reader
   ↓
Tree Builder
   ├── PageIndex tree if available/enabled
   └── fallback heading-based tree
   ↓
Clause-aware Chunker
   ↓
Index Docs
   ↓
Embeddings
   ↓
Azure AI Search
   ↓
Search-only RAG
```

For the Edison graph-enabled contract:

```text
Edison tree.json
   ↓
Normalized KG JSON
   ↓
Cosmos Gremlin Structural KG
   ↓
Legal Semantic Extraction
   ↓
Cosmos Gremlin Semantic KG
   ↓
Azure AI Search docs enriched with kgId
   ↓
Graph / Search / Hybrid GraphRAG
```

Final query flow:

```text
User question
   ↓
Query router
   ↓
Route selected:
   ├── graph  → direct Cosmos Gremlin semantic KG
   ├── search → Azure AI Search / structural retrieval
   └── hybrid → Azure AI Search → kgId → Cosmos Gremlin graph expansion
   ↓
Retrieval context
   ↓
Azure OpenAI answer generator
   ↓
Grounded answer with citations/source references
```

### 3.1 Graph-enabled contract

```text
Edison_NYPA_OandM_Contract_1
```

- Azure AI Search retrieval
- Cosmos Gremlin structural KG
- Cosmos Gremlin semantic legal KG

### 3.2 Uploaded contracts

Uploaded contracts through Streamlit currently support:

- Upload
- Parsing
- Tree generation
- Chunking
- Embeddings
- Azure AI Search upload
- Search-only RAG

Uploaded contracts do **not** currently support:

- Structural KG build
- Legal semantic extraction
- Cosmos Gremlin graph upsertion
- Graph-native route
- Hybrid graph expansion

If the user selects a non-graph-enabled contract and asks a graph/hybrid question, the system falls back to search.

```env
# Cosmos Gremlin

GREMLIN_DATABASE=<database-name>
GREMLIN_GRAPH=<graph-name>
```

Important:


Correct:

```text

```

Incorrect:

```text
https://<account>.documents.azure.com:443/
```

`AzureSearchTester` supports:

```python

```

### 10.3 `hybrid_search()`

Uses:

```text
search_text=query
vector_queries=[VectorizedQuery]
optional contract filter
optional graphReady filter
```

Used for normal semantic search and hybrid GraphRAG entry-point retrieval.

### 10.4 `retrieve_structural_scope()`

Generic structural retrieval for article/section/clause style questions.

Example:

```text
Summarize Article XII.
```

Detected as:

```json
{
  "structure_type": "Article",
  "identifier": "XII"
}
```

Retrieval uses:

```text
sectionTitle eq 'ARTICLE XII'
```

plus a post-filter to avoid parser bleed-over, such as a `13.1` clause incorrectly assigned to `ARTICLE XII`.

### 10.5 `list_contract_ids()`

Used by Streamlit sidebar to populate the contract dropdown.

### 10.6 `retrieve_contract_overview()`

Used for uploaded contracts when the question is like:

```text
Summarize this contract.
```

### 14.1 Main file


### 14.2 Purpose

Provides low-level Gremlin access for:

```text

```

This should be the preferred Gremlin client rather than creating separate global Gremlin clients in random modules.

### 14.3 Important principle

Do not allow LLMs to freely generate and execute arbitrary Gremlin queries.

Preferred production pattern:

```text
LLM QueryPlan JSON
→ validate plan against schema
→ compile to safe Gremlin query templates
→ execute
```

## 15. Azure AI Search ↔ Cosmos Gremlin Bridge

### 15.1 Problem

Cosmos Gremlin does not currently provide semantic vector search in this implementation.

Azure AI Search is the semantic/vector search layer.

Cosmos Gremlin is the symbolic graph traversal layer.

### 15.2 Solution

Azure AI Search documents are enriched with graph identifiers:

```text
kgId
parentKgId
graphReady
nodeType
graphLabel
```

Hybrid flow:

```text
Azure AI Search result
→ kgId
→ Cosmos Gremlin vertex
→ graph expansion
→ answer context
```

### 15.3 For Edison

```text
graphReady = true
kgId = clause/section vertex id
```

### 15.4 For uploaded contracts

```text
graphReady = false
kgId = empty
```

Uploaded contracts therefore use search-only RAG for now.

### 22.3 Important prompt principle

The answer generator should say:

```text

```

not:

```text

```

because hybrid and search routes include Azure AI Search text, metadata, and graph facts.

### 22.4 HTML unescape


### 24.7 Streamlit session-state caution

Do not modify a widget-bound key after widget instantiation.

Wrong pattern:

```python
question = st.text_area("Question", key="question_text")
st.session_state["question_text"] = "new value"
```

Correct pattern:

```python
def set_question_text(question):
    st.session_state["question_text"] = question
```

## 26. Current Known Issues / Limitations

### 26.1 Graph only exists for Edison

Uploaded contracts are search-only.

### 26.2 Semantic KG covers selected clauses only

Structural KG covers all Edison clauses, but semantic KG currently covers 20 selected clauses.

### 26.3 Router is keyword-based

The current router is sufficient for demo but should be replaced with LLM `QueryPlan` generation.

### 26.4 Party aliases are partially hardcoded

Current graph-native retriever uses hardcoded aliases for known parties.

Future approach:

```text
Build entity resolver dynamically from Party nodes, Defined Terms, and alias maps.
```

### 26.5 Article retrieval depends on parser quality

`retrieve_structural_scope()` includes a post-filter to handle parser bleed-over.

### 26.6 No graph embeddings

Cosmos Gremlin currently stores symbolic graph data, not embeddings.

Semantic retrieval is performed through Azure AI Search.

### 26.7 LLM should not write arbitrary Gremlin

Future automated query planning should output safe structured plans, not raw Gremlin.

## 30. Important Mental Model for Future AI Assistant

Do not think of the graph as the only knowledge source.

The system intentionally separates:

```text
Azure AI Search = semantic/full-text retrieval over contract text
Cosmos Gremlin = symbolic relationship traversal over legal facts
LLM = planner/answer synthesizer over retrieved context
```

For demo:

```text
graph route = direct structured facts
search route = text/structure retrieval
hybrid route = semantic search + graph expansion
```

For production:

```text
LLM QueryPlan
→ entity resolution
→ safe tool execution
→ evidence assembly
→ grounded answer
```

The LLM should not directly write arbitrary Gremlin. Instead, the LLM should output a validated structured plan that the system compiles to safe Gremlin/search calls.
